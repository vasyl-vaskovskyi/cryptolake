## Deviations review (Task A)

### D1 — Counter _total suffix registration
Verdict: accepted
Reasoning: This is a library-version constraint, not a design-intent change. Micrometer 1.13.4 on top of Prometheus Java client 1.x refuses register-time names ending in `_total`; the client re-appends the suffix at scrape time. Wire-parity (the only externally observable contract) is preserved and explicitly asserted in `WriterMetricsTest.messagesConsumed_counterName_exactlyWriter_messages_consumed_total` (scrape contains `writer_messages_consumed_total`). `NamingConvention.identity` is still applied per Tier 5 H4 at `WriterMetrics.java:51`.

### D2 — DatabaseConfig URL credential handling
Verdict: accepted
Reasoning: Caller-side adaptation forced by common's frozen API (`DatabaseConfig` is a single-field record `url()` — a Python-parity port). The JDBC `user`/`password` query-string form is a standard PostgreSQL driver feature that HikariCP forwards unmodified. Modifying `common/` (tagged `port-common-accepted`) would have been the wrong fix.

### D3 — MonitoringConfig port accessor
Verdict: accepted
Reasoning: Same caller-side adaptation pattern as D2. Common's `MonitoringConfig` has a single `prometheusPort` field (Python parity). Sharing one listener for `/ready` and `/metrics` is what the design §2.1 already describes — this is a rename, not a behavior change.

### D4 — LogInit.setLevel + logback.xml
Verdict: accepted
Reasoning: `logback.xml` loads declaratively from the classpath (Tier 2 §15 mandate); common exposes only `setLevel(String)`. Honouring an optional `LOG_LEVEL` env knob matches the Python shape. No invariant touched.

### D5 — LockSupport.parkNanos in retry backoff
Verdict: accepted
Reasoning: This is a Tier 2 rule-10 compliance fix, not a deviation in any substantive sense. `Thread.sleep` is forbidden in main sources by the skill's own rules (virtual-thread pinning). `LockSupport.parkNanos` at `StateManager.java:488` is the correct primitive, with an interrupt check at L489. Should have been in the original design; glad the developer noticed.

### D6 — RecordHandler MDC explicit try/finally
Verdict: accepted
Reasoning: Forced by common's `StructuredLogger.mdc()` signature (`AutoCloseable` with `close() throws Exception`). Using try-with-resources would have leaked a checked exception into `handle()`'s throws clause, which no call site can reasonably handle. The `try { ctx.close(); } catch (Exception ignored) {}` wrapper is a standard Java idiom for lambda-backed cleanup that provably cannot throw (`RecordHandler.java:128-168`). Identical MDC semantics are preserved.
