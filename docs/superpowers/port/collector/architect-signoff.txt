## Deviations assessment

- **2.1 Fixture format clarification (`.raw` is inner payload)**: acceptable — design §11 Q5 was hypothetical; investigation showed the simpler path is correct, byte identity is still verified end-to-end.
- **2.2 `KafkaProducerBridge` not `final`**: acceptable with note — sealed at the package level; a `KafkaProducerPort` interface is a reasonable future refactor (already noted in §5.1).
- **2.3 `GapReasons.VALID` additive update in common**: acceptable — additive change matches design §6.2 explicitly. Writer's `GapReasonsLocal` continues to work unchanged.
- **2.4 `DepthSnapshotResync.depthHandler` volatile setter**: acceptable — design §11 Q4 anticipated the circular dep; a `volatile` field with a single setter wired from `Main` is sound. `Main.java:198` calls `setDepthHandler` after construction on the main thread (visibility guaranteed by `volatile`).
- **2.5 `collector.txt` updated to 9-metric set**: acceptable — original fixture was incomplete; the new content matches the source-of-truth Python `metrics.py` and design §9.

## New issues found

- **Minor**: `Main.shutdownAll` order (`Main.java:362-410`) flushes the producer BEFORE `lifecycle.markCleanShutdown` — design §3.3 lists lifecycle after producer close, which is what was implemented; consistent with the design but I note that `markCleanShutdown` is called after `producer.close()`, which prevents any final lifecycle envelope from going through Kafka. Acceptable since lifecycle uses JDBC, not Kafka.
- **Minor**: `BackupChainReader.readLastDepthUpdateId` does `codec.readTree(rawNode.path("raw_text").asText().getBytes(UTF_8))` — re-encoding the string via `getBytes` works but allocates; not a correctness issue. (`backup/BackupChainReader.java:127-134`).
- **Minor**: `SubscriptionHandshake.QueueAckListener.waitForAck` discards non-matching ids without re-queueing (line 124 comment acknowledges) — given only one handshake at a time, this is fine.
- None of the §5 follow-ups are blockers in disguise: `KafkaProducerBridge` non-final, common-module `GapReasons` additive update, chaos test skeleton, `LifecycleStateManager` schema assumption, and harness simplification are all genuinely non-blocking.

## Recommendations (non-blocking)

- Future module: introduce `KafkaProducerPort` interface to restore `final` on `KafkaProducerBridge` and decouple test stubs (eliminates the package-private constructor footgun).
- `CollectorSession.create` (`CollectorSession.java:25-32`) constructs an unused `Instant now` at line 26-27 — can be removed; only `wallNow` is used.
- Consider adding a `_WS_STREAMS` capacity assertion at startup (per design §11 Q6) — currently absent; not load-bearing today but cheap insurance against a future stream/symbol expansion silently exceeding the 1024-stream Binance per-connection cap.
