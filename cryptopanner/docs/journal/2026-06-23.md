## Context / decisions
- **Seam over soak-only integration.** The genuinely live, two-connection-in-one-process part
  (real shadow socket, drop detection, promote hot-swap) sits behind `ShadowSession`. That makes
  the whole *orchestration* unit-testable with a fake while `LiveShadowSession` stays a thin,
  soak-validated impl — exactly the line §14.e draws, but now with the logic under test instead of
  untested integration code.
- **Aggregate equivalence, not per-pair.** A sealed overlap minute has many `(symbol,stream)`
  pairs; the conductor gates promote on ALL pairs passing (loops `EquivalenceChecker`), then merges
  all + logs one event. `RotationExecutor` stays the single-minute primitive; the multi-pair
  conductor merges via `OverlapMerger`+`DurableSegment` directly to keep all-or-nothing semantics.
- **Cross-process trigger = file.** Agent and collector are separate processes, so `/rotation/trigger`
  can't call the in-process manager — it drops `deploy/rotation-trigger`; the collector consumes it
  one-shot each minute. Mirrors active-slot / rotations.jsonl conventions.

## Session 3 — §11 observability gaps (the audit miss)

A challenge to "is the collector finished?" exposed that sessions 1–2 only audited §8 (capture)
and §5 (rotation) — never §11 (Health & observability), which is squarely collector code. Six real
gaps found and closed (all TDD), plus the honest finding that the soak harness doesn't exist yet.

- `9d34ca7` **§11.b heartbeat + §11.c metrics.** Collector touches
  `/tmp/cryptopanner-collector@<slot>.heartbeat` every tick (it touched *nothing* before — the agent
  would have seen it permanently down); `/metrics` gains `cryptopanner_rotation_events_total` +
  `cryptopanner_current_connection_age_seconds`.
- `b530bbf` **§10.d RECOVERED_AT_STARTUP.** `SegmentRecovery` now writes the rotation event when it
  merges/promotes leftover `.shadow` segments at startup (it was silently merging before).
- `a3840a8` **§11.a `StructuredLog`** (common) — JSON-Lines logger + `critical-events.jsonl` mirror.
- `dc54ca0` **§11.e events wired** — `ws_connect`/`ws_disconnect`/`ws_ping_missed`,
  `minute_sealed`/`late_frame_after_seal`, `rotation_started`/`verify_passed`/`verify_failed`/
  `promoted`/`forced_cutover`, threaded through clients/writers/manager/shadow. Verified live: the
  collector writes `logs/cryptopanner-collector@a.jsonl` in the §11.a shape.
- `8758ad3` **§11.c /status.rotation.** Collector publishes phase (IDLE/OVERLAP_VERIFYING/
  CUTOVER_PENDING) + connection age to a per-slot file (new `RotationStatus` codec); the Node Agent
  reads the active slot's file and renders the `rotation` object in `/status`.
- `67ffe36` **§15.f symbol-set re-read.** The rotation opener reloads `NodeConfig` at open time and
  rebuilds the shadow's sockets + writer set, so an operator symbol-set edit applies on the next
  rotation.

### Context / decisions
- **Why these were missed:** I had conflated "capture + rotation done" with "collector done." §11
  names the collector in every clause (per-slot logs, heartbeat, metrics, /status rotation fields,
  log events) — all collector obligations. Lesson recorded: audit *every* spec section that names
  the component, not just the feature areas you built.
- **Alerting stays the Monitor's job, but the *signals* are the collector's.** §13 alert *emission*
  is the Monitor (it scrapes /status+/metrics); the collector's duty is to expose the signals — now
  done (age gauge, rotation_events counter, rotation state, the §11.e events).

## Session 4 — Monitor module built end-to-end (§11.d, §13)

The last fully-unimplemented service. Built the whole cross-node Monitor in 10 TDD phases
(failing test → impl → green → commit), straight to `main`. 68 new tests; full reactor green.

- `MonitorConfig` extended (common) with the §15.c thresholds the record omitted
  (`upload_backlog_age`, `deploy_stuck`, REST poll-rate, `rest_rate_limit_persistence_hours`).
- **Pipeline** (`monitor` module): `StatusSnapshot`/`MetricsSnapshot` parsers → `NodeScraper`
  (`/status`+`/metrics` over HTTP) → `NodeStateTracker` + `AlertEvaluator` (the §13.a rules engine)
  → `AlertDispatcher` (§13.e dedup/correlation/recovery) → `WebhookAlertChannel` (Telegram/WhatsApp
  POST) → `RestartOrchestrator` + `AgentRestartClient` (§13.b/d gate, backoff, breaker, active-slot
  targeting, cliff exception) → `DeadMansSwitch` (§13.c) → `MonitorServer` (/dashboard, /api/nodes)
  → `MonitorCore` (one cycle) + `Main` (composition root, `start(cfg)` returns a closeable App).
- `config/dev/monitor.yaml` added so the compose `cryptopanner-monitor` service can boot.

### Context / decisions
- **Scrape /status *and* /metrics.** §11.c says the Monitor doesn't consume /metrics for history,
  but several §13.a conditions (upload backlog) have no /status signal — so the Monitor reads the
  *current* gauge values. The pure-log-event conditions (clock-skew degree, REST poll-rate,
  ws-disconnect duration, rotation-verify-fail counts) have **no remote signal** in today's
  contract; `AlertEvaluator` carries a `DEFERRED(§13.a)` marker for each so they slot in as new
  rules the moment a node-side signal lands. v1 fully implements every /status+/metrics-derivable
  condition.
- **`ShadowSession`-style seam for I/O.** `RestartClient` and the channels are interfaces, so the
  decision logic (backoff/breaker/gate) is unit-tested against fakes while the HTTP impls are
  exercised against an in-test `StubHttpServer` (stands in for a Node Agent / webhook). All timers
  are injected `Instant`/`Supplier<Instant>` — every cycle is deterministic.
- **Two-tier gate read.** `/status.state` already fuses systemd + heartbeat (§11.b), so the
  orchestrator treats `down`/`stuck` as restart-eligible and `running`/`degraded` as not (§13.d),
  with the cliff exception overriding for a degraded lock-holder.
