## Context / decisions
- **Seam over soak-only integration.** The genuinely live, two-connection-in-one-process part
  (real shadow socket, drop detection, promote hot-swap) sits behind `ShadowSession`. That makes
  the whole *orchestration* unit-testable with a fake while `LiveShadowSession` stays a thin,
  soak-validated impl — exactly the line §14.e draws, but now with the logic under test instead of
  untested integration code.
- **Aggregate equivalence, not per-pair.** A sealed overlap minute has many `(symbol,stream)`
  pairs; the conductor gates promote on ALL pairs passing (loops `EquivalenceChecker`), then merges
  all + logs one event. `RotationExecutor` stays the single-minute primitive; the multi-pair
  conductor merges via `OverlapMerger`+`DurableSegment` directly to keep all-or-nothing semantics.
- **Cross-process trigger = file.** Agent and collector are separate processes, so `/rotation/trigger`
  can't call the in-process manager — it drops `deploy/rotation-trigger`; the collector consumes it
  one-shot each minute. Mirrors active-slot / rotations.jsonl conventions.

- `6acaeab` **Item 2 — sequential-rotation primary hand-off.** `LiveShadowSession.promote()` hands
  itself off as the new primary via a new `PromotionSink` (after retiring the old primary). Collector

  *second* rotation acts on the promoted connection, not the retired one. Each rotation fully
  `close()`s the previously-promoted session, so there's no ticker/socket leak across daily
